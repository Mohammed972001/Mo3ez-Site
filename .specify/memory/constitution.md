# دستور المشروع · موكيت الصواري — متجر سعودي (Next.js, SEO-first)

> **النسخة:** 1.0.0 · **الحالة:** ✅ معتمد من المالك (2026-06-16)
> هذا الدستور هو المرجع الأعلى. أي خطة (plan) أو مواصفة (spec) أو تاسك (task) أو سطر كود
> يخالف مبدأً هنا يُرفض ويُعاد. عند أي تعارض، الأولوية بالترتيب:
> **SEO ← الأداء ← إمكانية الوصول ← مطابقة التصميم ← سرعة التسليم.**

---

## المهمة (Mission)
بناء موقع **موكيت/سجاد سعودي** على Next.js، هدفه الأول **التصدّر في نتائج البحث السعودي** للكلمات:
«موكيت»، «سجاد»، «موكيت الرياض»، «موكيت مساجد»، «موكيت مكتب»، «عشب صناعي»، «فينيل»، وما يماثلها —
مع تركيز جغرافي على **الرياض والمملكة العربية السعودية**. النموذج **كتالوج + توليد عملاء (lead-gen)**:
العرض للأرشفة والإقناع، والطلب عبر **واتساب / اتصال / فورم عرض سعر** (لا سلة/دفع في الإصدار الأول).

---

## المادة 1 — SEO أولاً (القاعدة العليا، غير قابلة للتنازل)
كل قرار تقني يُقيَّم أولاً بأثره على الأرشفة والترتيب. إلزاميًّا:
1. **عرض من الخادم (Server-Side Rendering / Static):** كل المحتوى القابل للأرشفة يُصدَّر عبر React Server Components — **ممنوع** إخفاء محتوى أساسي خلف JavaScript من جانب العميل.
2. **HTML دلالي:** `<h1>` واحد لكل صفحة، تدرّج عناوين صحيح، `<nav> <main> <article> <section> <footer>`، روابط `<a href>` حقيقية قابلة للزحف (لا أزرار تتنقّل بالـ JS).
3. **ميتاداتا لكل مسار:** عبر `generateMetadata` — `title`, `description` فريدة وعربية، `canonical`، OpenGraph، Twitter card، `lang="ar"` و`dir="rtl"`، و`hreflang="ar-sa"`.
4. **بيانات منظّمة (JSON-LD):** `Organization` + `LocalBusiness` (عنوان الرياض، ساعات العمل، هاتف، geo)، `Product`/`ItemList` لصفحات الكتالوج، `BreadcrumbList`، `Article` للمدوّنة، و`FAQPage` حيث يوجد أسئلة.
5. **خرائط وفهرسة:** `sitemap.ts` و`robots.ts` ديناميكيان يشملان كل التصنيفات والمنتجات والمقالات؛ canonical على كل صفحة؛ لا محتوى مكرّر.
6. **روابط نظيفة:** slugs عربية وصفية ثابتة — تُقرَّر مرة وتُثبَّت، ولا تتغيّر بدون redirect 301.
7. **محتوى عربي حقيقي:** كل تصنيف له وصف عربي أصيل غني بالكلمات المفتاحية، مع أقسام «لماذا نحن / المناطق التي نخدمها / أسئلة شائعة» لتقوية الـ topical authority المحلي.
8. **الصور للأرشفة:** أسماء ملفات وصفية بـ slug، و`alt` عربي وصفي لكل صورة، وأبعاد محدّدة.

## المادة 2 — موبايل أولاً (Mobile-First)
أغلب البحث من الجوال. التصميم يُبنى للجوال أولاً ثم يتوسّع. أهداف لمس ≥ 44px، خطوط مقروءة،
بلا تمرير أفقي، وبلا تفاعلات تعتمد على hover فقط. كل مكوّن يُختبر على عرض 360–390px قبل اعتماده.

## المادة 3 — الأداء و Core Web Vitals (ميزانية مُلزِمة)
كل صفحة لازم تحقّق على الجوال:
- **LCP < 2.5s** (هدف ≤ 2.0s) — عنصر الـ LCP يُحمَّل بأولوية (`priority`)، بلا حجب.
- **CLS < 0.1** (هدف ≤ 0.01) — أبعاد محجوزة لكل صورة/إطار، خطوط بـ `font-display:swap` + `size-adjust`، لا حقن محتوى يزيح التخطيط.
- **INP < 200ms** — JavaScript أدنى ما يمكن، تأجيل غير الحرج، لا حجب للـ main-thread.
قواعد داعمة: `next/image` (AVIF/WebP، `sizes` صحيحة، blur placeholder)، تقسيم الحزم،
استضافة الخطوط ذاتيًّا عبر `next/font`، CSS بلا runtime، وتفادي third-party scripts الثقيلة.

## المادة 4 — الهوية والتصميم (مطابقة 1:1 للموكاب المعتمد)
- **الصفحات المختارة:** الرئيسية **A** · السجاد/الكتالوج **B** · المنتج **A** · القائمة الكبرى **C**.
- **الألوان:** **Light = Palette 3 (Warm Artisan)** · **Dark = Palette 5 (Dark Premium)** — عبر CSS variables
  و`[data-theme]`/`dark:` + `prefers-color-scheme`، مع توكنز مشتقّة (success/sale/on-colors) بنفس منهج النظام.
- **الخطوط:** **Type 3** — العناوين: Fraunces (لاتيني) + ريم كوفي (عربي)؛ النص: Sora + Cairo. تُستضاف ذاتيًّا.
- **مصدر الحقيقة للمكوّنات:** نظام التصميم القائم (`components.css` + `store.css` + الأيقونات في `data.js`)
  يُنقَل بأمانة، RTL أصلي عبر logical properties، وتباين WCAG AA كحدّ أدنى.

## المادة 5 — المعمارية والتقنية
- **Next.js (أحدث مستقر) · App Router · React Server Components · TypeScript صارم.**
- **التنسيق: Tailwind CSS** — مع تعريف **توكنز التصميم كمتغيّرات CSS** ومطابقتها في `theme` الخاص بـ Tailwind،
  بحيث يبقى الـ light/dark والـ Palette 3/5 و Type 3 مضبوطين. Tailwind وقت-البناء = صفر runtime (يحمي الـ CLS).
  RTL عبر الخصائص المنطقية وإعداد `dir`؛ التزام صارم بـ `react-best-practices`.
- **البيانات:** ملفات داخل المشروع (TypeScript/JSON/MDX) للتصنيفات والمنتجات، تشير لصور `public/` — بلا قاعدة بيانات في v1، بطبقة بيانات معزولة (data layer) يسهل لاحقًا استبدالها بـ DB/CMS.
- **التحقّق:** مخططات `zod` لأي بيانات/فورم عند الحدود.
- **الطلب:** روابط واتساب (`wa.me`) + اتصال + فورم عرض سعر (server action، تحقّق zod، حماية spam).
- **i18n:** عربي فقط RTL الآن، والبنية جاهزة لإضافة لغة لاحقًا بلا إعادة هيكلة.

## المادة 6 — إمكانية الوصول (A11y · WCAG AA)
تباين AA، تركيز ظاهر (focus-visible)، ترتيب tab منطقي، ARIA حيث يلزم، يحترم `prefers-reduced-motion`،
ونصوص بديلة للصور. كل صفحة تجتاز فحص a11y قبل اعتمادها.

## المادة 7 — الجودة وبوابات السكيلز (مُفعّلة إلزاميًّا)
- **التصميم/الواجهة:** `frontend-design` + `web-design-guidelines`.
- **Next/React:** `nextjs` + `react-best-practices`.
- **مراجعة الكود:** `clean-code-guard` قبل إغلاق أي تاسك فيه كود إنتاجي.
- **الاختبارات:** `test-guard` لمراجعة الاختبارات؛ و`tdd` لمنطق البيانات/الـ SEO/الفورم.
- **التوثيق:** `docs-guard` لأي README/توثيق.
- **التفكير:** `systematic-debugging` لأي عطل، و`karpathy-guidelines` لإبقاء التغييرات جراحية.
- **الاختبار الآلي:** Playwright لمسارات حرجة، وفحص Lighthouse/CWV لكل صفحة رئيسية.

## المادة 8 — نظام المدوّنة (Blog · للـ SEO)
- مدوّنة **مبنية على ملفات (MDX داخل المستودع)** — لا back-end ولا CMS. المقالات جزء من الكود.
- يُبنى **الأساس والنظام الآن**: صفحة فهرس المدوّنة + صفحة المقال + التصنيفات/الوسوم + RSS اختياري،
  كل منها مُحسَّن للـ SEO (`Article` JSON-LD، breadcrumb، `generateMetadata`، صور `next/image`).
- **لا تُكتب مقالات إلا في نهاية المشروع**؛ عندها تُكتب عدّة مقالات يدويًّا داخل الكود لدعم الأرشفة
  (كلمات مفتاحية محلية: موكيت الرياض/مساجد/مكتب، أدلّة شراء، مقارنات…).

## المادة 9 — سير العمل الاحترافي و Git Flow
- نموذج فروع: **`main`** (إنتاج، محمي) · **`develop`** (تكامل) · **`feature/*`** · **`release/*`** · **`hotfix/*`**.
- **ممنوع** الـ commit المباشر على `main`. كل تاسك = فرع `feature/<task-id>-<slug>` من `develop` → Pull Request → دمج بعد المراجعة.
- **Conventional Commits** (feat/fix/chore/docs/refactor/test/perf…)، رسائل واضحة.
- **إسناد الـ commit:** بلا `Co-Authored-By`؛ يُختم بـ
  `Assisted-by: Claude (Anthropic) — review, planning & commit message (not code authorship)`.
- الـ push/PR على المستودع البعيد يتم بعد ربطه (المالك يوفّر الـ repo والصلاحيات).
- كل تاسك يُغلق فقط بعد: بناء ناجح + اجتياز معايير القبول + مراجعة السكيل المناسب.

## المادة 10 — سجلّ القرارات التعليمي (عربي · RTL)
- يُحدَّث ملف `docs/seo-architecture-log-ar.md` بعد **كل قرار** تقني/معماري/SEO مؤثّر.
- بالعربية RTL، الكلمات الإنجليزية بين أقواس وسط السطر لسهولة القراءة، وبأسلوب **تعليمي** يشرح
  «ماذا قرّرنا ولماذا وأثره على الـ SEO» — ليكون مرجعًا يتعلّم منه المالك الـ SEO بعمق.

## المادة 11 — منهجية العمل (Spec-Driven, بالترتيب, تحقّق 100%)
1. التسلسل: **الدستور ← المواصفات (spec) ← الخطة (plan) ← التاسكات (tasks) ← التنفيذ (implement)**.
2. التاسكات تُنفَّذ **بالترتيب**؛ ما يُفتَح تاسك جديد إلا بعد إغلاق السابق.
3. كل تاسك له **معايير قبول صريحة**: مطابقة التصميم + اجتياز CWV + متطلبات SEO + بوابة السكيل المناسب + سجلّ القرارات محدَّث.
4. «مكتمل» = اتعمل، **اتحقّق منه فعليًّا** (تشغيل/قياس/فحص)، ووُثِّق. لا «مكتمل» بالادّعاء.
5. أي تعديل على الدستور يرفع النسخة ويُسجَّل سببه ويُعتمد من المالك.
