# 📘 سجلّ القرارات المعمارية والـ SEO — ملف تعليمي

> ملف يشرح **كل قرار تقني ومعماري** اتّخذناه في بناء الموقع، ولماذا، وأثره على الـ SEO —
> مكتوب ليكون مرجعًا تتعلّم منه الـ SEO بعمق. يُحدَّث بعد كل قرار مؤثّر (المادة 10 من الدستور).
> مرتّب زمنيًّا — القرارات الجديدة تُضاف في الأسفل.

---

## القرار 0 — حوكمة المشروع: لماذا «دستور» و«سجلّ قرارات» قبل أي كود؟

**ما القرار:** كتبنا دستورًا (Constitution) يحدّد المبادئ غير القابلة للتنازل، ثم بدأنا هذا السجلّ، **قبل** كتابة أي كود.

**لماذا:** المواقع التي تتصدّر البحث لا تُبنى عشوائيًّا. عندما تكون القاعدة العليا واضحة («SEO أولاً»)، كل قرار لاحق يُقاس عليها فلا نتشتّت. وهذا أسلوب الشركات الاحترافي (Spec-Driven Development): تُعرّف القواعد، ثم المواصفات، ثم الخطة، ثم التنفيذ.

**الأثر على الـ SEO:** الاتساق. مثلًا قاعدة «كل محتوى يُعرض من الخادم» لو لم تُكتب كمبدأ، يسهل أن يخالفها أحد المكوّنات لاحقًا فيضيع محتوى من الأرشفة.

---

## القرار 1 — لماذا Next.js + مكوّنات الخادم (React Server Components)؟

**ما القرار:** استخدام **Next.js (App Router)** مع **مكوّنات الخادم (Server Components)** كأساس، وعرض كل المحتوى المهم من الخادم (Server-Side Rendering / Static Generation).

**لماذا:** عناكب البحث (Crawlers) تقرأ الـ HTML الذي يصل من الخادم. لو كان المحتوى يُبنى في المتصفّح بالـ JavaScript فقط (Client-Side Rendering)، فقد يصل للعنكبوت صفحة شبه فارغة فلا يؤرشف منتجاتك. مع مكوّنات الخادم، المنتجات والأوصاف تصل **جاهزة في الـ HTML** من أول طلب.

**الأثر على الـ SEO:**
- أرشفة كاملة وفورية للمحتوى (الكلمات المفتاحية موجودة في مصدر الصفحة).
- أسرع ظهور للمحتوى (First Contentful Paint) لأن المتصفّح يستلم HTML جاهزًا.
- يدعم التوليد الثابت (Static) والـ (ISR) فتُخدَم الصفحات من الـ CDN بسرعة عالية.

---

## القرار 2 — لماذا Tailwind CSS مع توكنز متغيّرات CSS؟

**ما القرار:** التنسيق بـ **Tailwind CSS**، لكن مع تعريف **ألوان وخطوط التصميم كمتغيّرات CSS (CSS Variables)** ثم ربطها بـ `theme` الخاص بـ Tailwind.

**لماذا:** Tailwind يولّد الـ CSS **وقت البناء (build-time)** فقط — لا يوجد JavaScript يحسب الأنماط في المتصفّح (لا runtime). هذا يحمي مؤشّر **(CLS)** لأن الأنماط جاهزة فورًا ولا «تقفز» العناصر. وربط الألوان بمتغيّرات CSS يجعل تبديل **الوضع الفاتح/الداكن (Light/Dark)** فوريًّا ونظيفًا: نغيّر قيمة المتغيّر فيتبدّل الموقع كله.

**الأثر على الـ SEO/الأداء:**
- **CLS منخفض:** لا حقن أنماط متأخّر يزيح التخطيط.
- **حجم أصغر:** Tailwind يحذف الأنماط غير المستخدمة، فتقلّ بايتات الـ CSS → أسرع تحميل.
- دعم نظيف للـ Dark Mode بلا تكلفة أداء.

**التطبيق هنا:** Light = Palette 3 (Warm Artisan) · Dark = Palette 5 (Dark Premium) · الخطوط Type 3 (Fraunces + ريم كوفي للعناوين، Sora + Cairo للنص).

---

## القرار 3 — موبايل أولاً (Mobile-First)

**ما القرار:** تصميم وتطوير كل شيء **للجوال أولًا** (360–390px) ثم التوسّع للشاشات الأكبر.

**لماذا:** جوجل يعتمد **فهرسة الجوال أولًا (Mobile-First Indexing)** — أي يقيّم نسخة الجوال من موقعك للترتيب. وأغلب عمليات البحث عن «موكيت الرياض» تتم من الجوال. لو كانت تجربة الجوال ضعيفة، يتأثّر ترتيبك حتى على الديسكتوب.

**الأثر على الـ SEO:** ترتيب أفضل (لأن جوجل يقيس الجوال)، وتجربة استخدام أعلى → معدّل ارتداد أقل → إشارات إيجابية.

---

## القرار 4 — Core Web Vitals: الميزانية وكيف نحقّقها

**ما القرار:** ميزانية مُلزِمة لكل صفحة: **LCP < 2.5s** · **CLS < 0.1** (هدف 0.01) · **INP < 200ms**.

**لماذا:** هذه مؤشّرات تجربة الصفحة (Page Experience) وهي **عامل ترتيب فعلي** في جوجل:
- **LCP (Largest Contentful Paint):** زمن ظهور أكبر عنصر مرئي (غالبًا صورة البطل). نحمّله بأولوية (`priority`).
- **CLS (Cumulative Layout Shift):** مقدار «قفز» العناصر. نمنعه بحجز أبعاد كل صورة/إطار مسبقًا.
- **INP (Interaction to Next Paint):** سرعة استجابة الموقع للنقر/اللمس. نقلّل الـ JavaScript ونؤجّل غير الحرج.

**الأثر على الـ SEO:** صفحات سريعة ومستقرّة = ترتيب أعلى + رضا مستخدم أعلى.

---

## القرار 5 — استراتيجية الصور (الأثقل في الأداء والأغنى في الـ SEO)

**ما القرار:** كل الصور عبر **`next/image`** بصيغ حديثة (AVIF/WebP)، بأبعاد محدّدة و`sizes` صحيحة وplaceholder ضبابي، **وإعادة تسميتها بـ slugs عربية وصفية** مع **نص بديل (alt) عربي** لكل صورة.

**لماذا:** الصور الحالية ملفات WhatsApp ثقيلة بأسماء توقيت (مثل `WhatsApp Image ...`) — سيّئة للأداء وللأرشفة. الصور غالبًا هي عنصر الـ LCP، فتحسينها يحسّن السرعة مباشرة. واسم الملف والـ alt الوصفيان يساعدان في **أرشفة الصور (Google Images)** وهي مصدر زيارات مهم لمنتجات بصرية كالموكيت.

**الأثر على الـ SEO:**
- LCP أسرع (صور أخف بصيغ حديثة).
- ظهور في بحث صور جوجل بكلمات مثل «موكيت مساجد».
- لا CLS من الصور (أبعاد محجوزة).

---

## القرار 6 — HTML الدلالي + الميتاداتا + البيانات المنظّمة (JSON-LD)

**ما القرار:** بنية HTML دلالية (`<main> <article> <nav>` و`<h1>` واحد)، و`generateMetadata` لكل مسار، و**بيانات منظّمة JSON-LD**: `LocalBusiness` (للرياض)، `Product`/`ItemList`، `BreadcrumbList`، `Article`، `FAQPage`.

**لماذا:** الـ HTML الدلالي يساعد العنكبوت على فهم بنية الصفحة وأهميّتها. والبيانات المنظّمة تتيح **النتائج الغنية (Rich Results)** (نجوم تقييم، أسعار، مسار تنقّل) فيزيد معدّل النقر. و`LocalBusiness` بعنوان الرياض يقوّي **السيو المحلي (Local SEO)** — جوهري لأنك تستهدف «الرياض».

**الأثر على الـ SEO:** فهم أفضل من جوجل + نتائج غنية تلفت النظر + قوة محلية في نتائج الرياض والخرائط.

---

## القرار 7 — sitemap و robots و canonical والروابط النظيفة

**ما القرار:** `sitemap.ts` و`robots.ts` ديناميكيان يشملان كل التصنيفات والمنتجات والمقالات، ورابط canonical لكل صفحة، وslugs عربية ثابتة لا تتغيّر بدون redirect 301.

**لماذا:** الـ sitemap يدلّ جوجل على كل صفحاتك ليؤرشفها. الـ canonical يمنع مشكلة **المحتوى المكرّر (Duplicate Content)**. والروابط الثابتة تحفظ ترتيبك (تغيير الرابط بلا 301 = فقدان ترتيب الصفحة القديمة).

**الأثر على الـ SEO:** فهرسة كاملة، لا تشتيت لقوّة الصفحات، وحفظ الترتيب عبر الزمن.

---

## القرار 8 — بيانات مبنية على ملفات (File-Based) لا قاعدة بيانات

**ما القرار:** التصنيفات والمنتجات كملفات (TypeScript/JSON) داخل المشروع تشير لصور `public/`، مع **طبقة بيانات معزولة** يسهل لاحقًا استبدالها بقاعدة بيانات أو CMS.

**لماذا:** للكتالوج الثابت، الملفات تعني صفحات يمكن **توليدها ثابتًا (Static)** وخدمتها من الـ CDN بأقصى سرعة وأقل تكلفة — مثالي للـ SEO. والعزل يحمينا: لو احتجنا لاحقًا لوحة تحكم/مخزون ديناميكي، نبدّل الطبقة دون إعادة بناء الواجهة.

**الأثر على الـ SEO:** أقصى سرعة (صفحات ثابتة) + بساطة = استقرار وأداء عالٍ.

---

## القرار 9 — نظام مدوّنة (Blog) مبني على MDX للـ SEO

**ما القرار:** بناء **نظام مدوّنة من ملفات MDX** داخل الكود (لا back-end): فهرس + صفحة مقال + تصنيفات/وسوم، كلها مُحسَّنة (`Article` JSON-LD، breadcrumb، metadata). **الأساس يُبنى الآن، والمقالات تُكتب في النهاية.**

**لماذا:** المدوّنة تنشئ صفحات تستهدف **كلمات بحث طويلة (Long-tail keywords)** مثل «أفضل أنواع موكيت المساجد» أو «الفرق بين الموكيت والفينيل» — هذه تجلب زيارات مؤهّلة وتبني **السلطة الموضوعية (Topical Authority)** التي ترفع ترتيب الموقع كله. وكونها داخل الكود = صفحات ثابتة سريعة ومؤرشفة جيدًا.

**الأثر على الـ SEO:** مصدر زيارات متجدّد، تقوية للكلمات الرئيسية، وروابط داخلية تدعم صفحات المنتجات.

---

## القرار 10 — سير العمل الاحترافي (Git Flow)

**ما القرار:** نموذج فروع `main`/`develop`/`feature/*`، Conventional Commits، Pull Requests، ممنوع الـ commit المباشر على `main`، وكل تاسك على فرعه.

**لماذا:** ليس له أثر مباشر على الـ SEO، لكنه **أثر مباشر على الجودة والاستقرار**: تغييرات معزولة وقابلة للمراجعة والتراجع، فلا «نكسر» موقعًا مؤرشفًا بتعديل عشوائي. الاستقرار = إشارة جودة غير مباشرة.

**الأثر على الـ SEO (غير مباشر):** حماية الموقع المنشور من الأعطال، وتاريخ تغييرات نظيف.

---

## القرار 11 — تبنّي GEO (تحسين محرّكات التوليد) إلى جانب السيو

**ما القرار:** نُحسّن الموقع ليس فقط لجوجل، بل أيضًا ليُقتبس داخل إجابات الذكاء الاصطناعي (ChatGPT، Google AI Overviews، Perplexity، Gemini) — عبر كتل إجابة مباشرة، أسئلة شائعة بـ `FAQPage` schema، أرقام/حقائق واضحة، قوائم وجداول، وملف `llms.txt`.

**لماذا:** الناس بدأت تسأل الـ AI مباشرة («أفضل موكيت بالرياض؟») بدل تصفّح روابط جوجل. من يظهر في إجابة الـ AI يكسب العميل قبل المنافس. والتداخل بين نتائج جوجل ومصادر الـ AI صار أقل من 20% — فهي قناة منفصلة يجب كسبها.

**الأثر على الـ SEO/الأعمال:** قناة اكتساب جديدة (AI)، وثقة أعلى (الاقتباس = توصية)، وتميّز عن منافسين لا يهتمون بها بعد. التفاصيل الكاملة في `docs/seo-geo-strategy-ar.md`.

---

## القرار 12 — السيو المحلي و توحيد NAP (استهداف «الرياض»)

**ما القرار:** نعتمد بيانات نشاط موحّدة (NAP = الاسم/العنوان/الهاتف) تظهر بنفس الصيغة في الموقع و`LocalBusiness` JSON-LD، مع `areaServed` لمناطق الخدمة، وبنية جاهزة لصفحات المناطق والتقييمات. الهاتف `+966546465316`، العنوان: الرياض — العزيزية — حراج بن قاسم، 24/7 (الاسم النهائي قيد الحسم).

**لماذا:** عندما تستهدف مدينة، السيو المحلي هو المعركة الأساسية. تطابق NAP عبر كل المنصّات إشارة ثقة قوية لجوجل لربطك بـ«الرياض». و`LocalBusiness` schema + صفحات المناطق ترفعك في النتائج المحلية وخرائط جوجل.

**الأثر على الـ SEO:** ظهور أقوى في بحث «موكيت الرياض» وما يشبهه، وفي الباقة المحلية (Local Pack) والخرائط — وهي أعلى النتائج نية شرائية.

## القرار 13 — تثبيت spec-kit الرسمي + تهيئة المشروع (T00)

**ما القرار:** اعتمدنا أداة **spec-kit الرسمية** (`specify`) فعليًّا — لا مجرد محاكاة يدوية — فأضافت الهيكل القياسي (`.specify/templates`, `scripts`, `workflows`) وأوامر Claude (`/speckit-constitution|specify|plan|tasks|implement`). الدستور والمواصفات المعتمدة بقيت كما هي. ثم نفّذنا **T00**: تهيئة **Next.js 16 + React 19 + Tailwind v4 + TypeScript صارم**، layout عربي `lang=ar dir=rtl`، هيكل المجلدات، وملف `business.ts` كمصدر موحّد لبيانات NAP.

**لماذا:** الالتزام بمنهجية spec-kit القياسية (Spec-Driven Development) يضمن أن كل ميزة تمرّ بـ: مواصفة ← خطة ← تاسكات ← تنفيذ، بتتبّع واضح ومعايير قبول — وهو «شغل الشركات الاحترافي» المطلوب. أمّا اختيار Next.js (RSC) + Tailwind (صفر runtime) فيخدم الـ SEO والأداء مباشرة (عرض من الخادم + CLS منخفض).

**الأثر على الـ SEO:** الأساس التقني (SSR + بنية نظيفة) جاهز لتطبيق كل قواعد المادة 1؛ و`lang=ar dir=rtl` من الجذر إشارة صحيحة للزواحف من اليوم الأول.

## القرار 14 — نظام الثيم بالتوكنز (لايت/دارك) والخطوط الذاتية (T01)

**ما القرار:** بنينا الثيم بمتغيّرات CSS: `:root` = Palette 3 (Warm Artisan) للوضع الفاتح، و`[data-theme="dark"]` = Palette 5 (Dark Premium) للداكن، مربوطة في Tailwind v4 عبر `@theme inline`. التبديل بسمة على `<html>` مع سكربت no-flash يحترم `prefers-color-scheme`. الخطوط (Type 3) تُستضاف ذاتيًّا عبر `next/font` مع `display:swap`. كل أزواج الألوان متحقّقة WCAG AA رياضيًّا قبل الكتابة.

**لماذا:** (1) متغيّرات CSS = تبديل ثيم بصفر إعادة-حساب تخطيط (CLS=0). (2) `next/font` يستضيف الخطوط ذاتيًّا ويحقن مقاييس fallback → يمنع وميض الخط وحجز المساحة (يحمي LCP/CLS — المادة 3). (3) Tailwind وقت-البناء = صفر CSS runtime. (4) تحقّق التباين مسبقًا يضمن a11y (المادة 6) من الأساس.

**الأثر على الـ SEO:** أداء أفضل (LCP/CLS) = إشارة ترتيب مباشرة؛ ووضع داكن سليم + تباين AA يحسّنان تجربة المستخدم ومقاييس التفاعل التي تؤثّر غير مباشرة على الترتيب.
